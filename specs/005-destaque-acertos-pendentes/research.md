# Phase 0 Research: Destaque de acertos do concurso mais recente nos cards de teimosinha

Nenhum `NEEDS CLARIFICATION` restou no Technical Context do plan — as decisões abaixo documentam o raciocínio técnico e as alternativas descartadas para os pontos que o spec deixou como "o quê", não "como".

## 1. Encontrar o "concurso mais recente já apurado" sem nova consulta

- **Decision**: `JogoService.listarComResumo()` já chama `conferirConcursos(jogo)` (retorna `List<ConferenciaConcurso>` ordenada ascendentemente por número de concurso, de `concursoInicial` a `concursoFinal`) só para montar `resumo`/`ganhoTotal`. Uma nova função privada, ex. `dezenasAcertadasUltimoConcurso(List<ConferenciaConcurso> concursos, ResumoJogo resumo)`, percorre essa mesma lista **de trás para frente** e retorna `dezenasAcertadas` do primeiro item cuja `situacao` não seja `ConferenciaConcurso.PENDENTE`. Se nenhum for encontrado (todos pendentes) ou se `resumo.getPendentes() == 0`, retorna `List.of()`.
- **Rationale**: a varredura de trás para frente encontra o maior número de concurso com resultado importado independentemente de eventuais furos na sequência (não depende de os concursos pendentes formarem uma cauda contígua); reaproveita 100% os dados que `listarComResumo()` já calcula hoje, sem nenhuma chamada nova ao Mongo ou à API da Caixa.
- **Alternatives considered**: nova query dedicada (`ConcursoRepository.findTopByLoteriaAndConcursoLessThanEqualOrderByConcursoDesc`) — descartada por duplicar dado que já está em memória na mesma requisição; percorrer a lista de frente para trás até achar o primeiro `PENDENTE` e pegar o anterior — descartada por assumir que todos os concursos pendentes formam uma cauda contígua no fim do intervalo, o que é verdade na prática (import é sequencial) mas não é uma garantia do modelo de dados — a varredura reversa é igualmente simples e não depende dessa suposição.

## 2. Onde expor o dado: novo campo em `JogoComResumo`

- **Decision**: adicionar `List<Integer> dezenasAcertadasUltimoConcurso` a `JogoComResumo`, populado por `listarComResumo()` chamando a função da decisão 1. Vazio (`List.of()`) cobre igualmente os dois casos do spec em que nada deve ser destacado: teimosinha expirada (`pendentes == 0`, FR-004) e teimosinha sem nenhum concurso apurado ainda (FR-005) — o JSP não precisa distinguir os dois casos, só verificar pertencimento à lista.
- **Rationale**: `JogoComResumo` já é o view model consumido tanto por `jogos.jsp` quanto por `GET /api/jogos` (`JogoRestController.listar()`); adicionar um campo aqui evita criar um DTO novo ou uma segunda chamada do cliente.
- **Alternatives considered**: expor só o número do concurso mais recente e recalcular os acertos no JS (o cliente já tem `jogo.numeros`, bastaria buscar `dezenasSorteadas` desse concurso) — descartado porque exigiria uma chamada adicional por card (`GET /api/loterias/{loteria}/concursos/{numero}`) só para obter algo que o servidor já calculou de graça durante `conferirConcursos`; calcular no JSP diretamente sobre `resumo` — não é possível, porque `resumo` (contagens) não carrega as dezenas do concurso, só os totais.

## 3. Cor do destaque: reaproveitar `.dezena.acertada` (recolorida) em vez de criar uma classe nova

- **Decision**: em vez de introduzir uma classe nova para o destaque nas dezenas grandes do card, a classe já existente `.dezena.acertada` (hoje `background: #166534; color: #fff;`, usada na coluna "Dezenas jogadas/sorteadas (acertos destacados)" do painel de sorteios premiados e de `/jogos/{id}`) é recolorida para o mesmo par de cores de `.badge.premiado` (`background: #dcfce7; color: #166534;`). `jogos.jsp` passa a aplicar essa mesma classe `acertada` aos `<span class="dezena grande">` cujo número esteja em `item.dezenasAcertadasUltimoConcurso`, com o padrão condicional já usado em `jogo.jsp` (`${item.dezenasAcertadas.contains(numero) ? 'acertada' : ''}`).
- **Rationale**: resolve as duas User Stories com uma única mudança de CSS — FR-003 (cor do destaque no card = cor do badge premiado) e FR-007 (suavizar o verde da coluna "acertos destacados") acabam pedindo exatamente a mesma cor final, e a Clarification já autorizou suavizar `.dezena.acertada` globalmente (afetando também `/jogos/{id}`). Criar uma classe nova só para o card duplicaria uma regra de cor idêntica.
- **Alternatives considered**: nova classe `.dezena.destaque-recente` com as cores do badge premiado, mantendo `.dezena.acertada` no tom escuro atual — descartada após a Clarification confirmar que a suavização de `.dezena.acertada` é aceita globalmente; manter duas classes com a mesma cor final não traria nenhum benefício e ainda deixaria a coluna "acertos destacados" com o verde escuro que o usuário pediu para suavizar.

## 4. Nenhuma mudança na regra de premiação/pendência

- **Decision**: `ResumoJogo.getPendentes()` (já calculado por `resumo(concursos)`) é reaproveitado tal como está para decidir se o destaque deve aparecer — nenhuma mudança em `resumo(...)`, `conferirConcursos(...)` ou `Loteria.premiado(...)`.
- **Rationale**: a definição de "teimosinha com concursos pendentes" já existe e é exatamente a mesma usada hoje pelo badge "pendente" no card — reaproveitá-la mantém as duas UIs (contagem do badge e presença do destaque) sempre consistentes entre si sem nenhuma lógica duplicada.
- **Alternatives considered**: calcular "ativa vs. expirada" comparando `concursoFinal` com o maior concurso importado na base (`ConcursoRepository`) — descartada por exigir uma consulta adicional por loteria que o `resumo.pendentes` já responde sem custo extra.
